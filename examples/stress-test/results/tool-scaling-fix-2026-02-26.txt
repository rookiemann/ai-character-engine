================================================================================
BOTTLENECK ANALYSIS — WHY 32 CHARS SATURATES
================================================================================

The RTX 3090 hits its token generation ceiling at ~2600-5800 tok/s with GGUF
format. At 32 concurrent agents queuing every 500ms, requests pile up faster
than the GPU can drain them, causing:
  - Average latency balloons to 55s (mostly queue wait time, not compute)
  - 42 requests still in-flight at shutdown
  - Throughput *drops* from 4.76/s (16c) to 2.00/s (32c) — classic saturation

The bottleneck is NOT the engine, NOT the scheduler, NOT the network.
It is purely GPU token generation throughput with GGUF format.

See vLLM optimization recommendations below.

─── FIX 1: SWITCH FROM GGUF TO FP16 SAFETENSORS (CRITICAL — 5-8x throughput) ──

GGUF in vLLM is "highly experimental and under-optimized" per official docs.
Benchmarks show GGUF at 93 tok/s vs Marlin-AWQ at 741 tok/s — 8x difference.
GGUF was designed for llama.cpp CPU pipelines, not vLLM's GPU-first architecture.
vLLM dequantizes GGUF on-the-fly, losing all GPU-native kernel advantages.

xLAM-2-1B is only ~2GB in FP16 — trivially fits on 24GB with massive KV room.

Options (best to worst):
  a) Use FP16 safetensors directly: --model Salesforce/xLAM-2-1b-fc-r
  b) AWQ 4-bit with Marlin kernels (if you want max KV cache space)
  c) GPTQ 4-bit with Marlin kernels

Expected impact: 5-8x throughput improvement alone.
This single change dwarfs all others combined.

─── FIX 2: REMOVE --enforce-eager (HIGH — 20-40% throughput gain) ────────────

--enforce-eager disables CUDA graph capture, forcing eager execution for every
forward pass. CUDA graphs pre-record GPU kernel sequences and replay with
near-zero CPU launch overhead. For a 1B model where compute per step is small,
this CPU overhead is proportionally large.

Remove the flag entirely. Default mode uses CUDA graphs automatically.
Memory cost is trivial with a 941MB model on 24GB.

─── FIX 3: RAISE --gpu-memory-utilization to 0.95 (HIGH — 38% more KV cache) ─

Currently 0.7 = 16.8GB allocated. With a 941MB model, ~15.8GB for KV cache.
At 0.95 = 22.8GB allocated, ~21.8GB for KV cache = 38% more concurrent seqs.

Default in vLLM is 0.9. RTX 3090 benchmarks commonly use 0.90-0.95.

─── FIX 4: ENABLE PREFIX CACHING (HIGH for this workload — 2-5x TTFT cut) ────

Agent prompts share long common prefixes: system prompt, character identity,
tool definitions. Without APC, every request recomputes full prefill.
With APC, shared prefix computed once and reused from KV cache.

Real-world: TTFT drops from 4.3s to 0.6s on shared prefixes.
For ~800 token prompts with ~400-600 shared tokens: 50-75% prefill reduction.

Add to launcher: --enable-prefix-caching
In code: llm_kwargs["enable_prefix_caching"] = True

─── FIX 5: REDUCE --max-model-len to 4096 (MEDIUM-HIGH — 2x concurrency) ────

KV cache per sequence pre-allocated for max length. Current: 8192 per seq.
Actual usage: ~950 tokens (active), ~500 (background), ~330 (dormant).
Reserving 8192 wastes ~88% of each allocation.

Set --max-model-len 4096 (or 2048 if max actual usage stays under ~1500).
This doubles the number of sequences that fit in KV cache memory.

─── FIX 6: ADD --num-scheduler-steps 10 (MEDIUM — 15-30% throughput gain) ────

Default: 1 step between GPU-CPU syncs. For a small 1B model where each
forward pass is very fast, CPU scheduling overhead is a large fraction of
total time. Multi-step scheduling executes 10 decode steps before syncing.

Add to launcher: --num-scheduler-steps 10

─── FIX 7: RAISE --max-num-seqs to 128 (MEDIUM — higher ceiling) ─────────────

With fixes 3+5, KV cache can hold many more sequences. Raise the ceiling:
  --max-num-seqs 128
Cost is zero if actual load is lower — vLLM dynamically allocates.

─── FIX 8: REDUCE LAUNCHER POLLING TO 1ms (MEDIUM — lower per-token latency) ─

In vllm_launcher.py engine_step_loop: change asyncio.sleep(0.005) to 0.001.
Also: always step engine regardless of pending count (step() is no-op when empty).

─── NOT RECOMMENDED ───────────────────────────────────────────────────────────

  - Speculative decoding: Model already 1B, no meaningful smaller draft exists
  - FP8 KV cache: RTX 3090 is compute capability 8.6, needs 8.9+ (Ada/Hopper)
  - Tensor parallelism 3090+3060: Slower GPU bottlenecks, PCIe overhead,
    1B model doesn't need it. Keep 3060 for embeddings.
  - Chunked prefill tuning: Marginal for short prompts (~800 tokens)
