# AI Character Engine -- Memory System

## Overview

The memory system gives each character a believable sense of history. It uses a 3-tier architecture inspired by human memory: short-term working memory, medium-term episodic memory with natural fading, and long-term LLM-compressed summaries.

All memory is scoped per `(characterId, playerId)` pair, enabling multi-player support where each player has a distinct relationship history with each character.

---

## 3-Tier Memory Architecture

```
+-------------------------------------------------------+
|                   WORKING MEMORY                       |
|  Ring buffer (most recent N interactions)              |
|  Always included in LLM context                        |
|  Size: workingMemorySize (default 5)                   |
+-------------------------------------------------------+
                        |
                        | overflow (oldest entries fall off)
                        v
+-------------------------------------------------------+
|                  EPISODIC MEMORY                       |
|  Importance-scored records (1-10)                      |
|  Fading: currentImportance decays over time            |
|  Types: observation, interaction, reflection, dialogue |
|  Tags, event types for retrieval                       |
|  Retrieved by recency + importance for LLM context     |
+-------------------------------------------------------+
                        |
                        | periodic LLM compression
                        v
+-------------------------------------------------------+
|                 CHARACTER SUMMARY                      |
|  LLM-compressed narrative of character's history       |
|  Includes: summary text, relationship notes, key facts |
|  Auto-regenerated every N decisions                    |
|  Versioned (monotonically increasing)                  |
+-------------------------------------------------------+
```

### Working Memory

**File:** `src/memory/WorkingMemory.ts`

Working memory is a fixed-size ring buffer that holds the most recent interactions. It functions like a conversation window, providing immediate context to the LLM.

- **Structure:** Each entry has `role` (user/assistant/system), `content`, and `turnIndex`
- **Size:** Configurable via `workingMemorySize` (default: 5 entries)
- **Behavior:** When the buffer is full, the oldest entry is dropped
- **Usage:** Always included in the LLM prompt as recent conversation context

```typescript
interface WorkingMemoryEntry {
  id: string;
  characterId: string;
  playerId: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  turnIndex: number;
  createdAt: number;
}
```

### Episodic Memory

**File:** `src/memory/EpisodicMemory.ts`

Episodic memories are individual records of events, interactions, observations, and reflections. Each memory carries an importance score that fades over time, simulating natural forgetting.

```typescript
interface MemoryRecord {
  id: string;
  characterId: string;
  playerId: string;
  type: 'observation' | 'interaction' | 'reflection' | 'dialogue';
  content: string;           // Full text of the memory
  summary: string;           // Short summary for compact retrieval
  importance: number;        // Base importance (1-10, set at creation)
  currentImportance: number; // Fading value (decays over time)
  isDeep: boolean;           // Resists fading (importance >= 9)
  isPermanent: boolean;      // Never decays or prunes (importance 10, trauma)
  tags: string[];            // For tag-based clustering/retrieval
  eventType?: string;        // Source event type
  decayRate: number;         // Multiplier for decay speed
  createdAt: number;
  lastAccessedAt: number;
}
```

**Memory Types:**
- `observation` -- Things the character noticed (events, environment changes)
- `interaction` -- Direct interactions with the player or other characters
- `reflection` -- Internal thoughts or conclusions drawn by the character
- `dialogue` -- Conversation content (chat, multi-agent conversations)

### Character Summary

**File:** `src/memory/CharacterSummary.ts`

The character summary is a compressed narrative generated by the LLM from the character's episodic memories. It provides long-term context without consuming excessive token budget.

```typescript
interface CharacterSummaryRecord {
  id: string;
  characterId: string;
  playerId: string;
  summary: string;           // Narrative summary text
  relationshipNotes: string; // How the character feels about the player
  keyFacts: string[];        // Important facts as bullet points
  version: number;           // Incremented on each regeneration
  generatedAt: number;
}
```

## Memory Fading

Memory fading simulates natural forgetting. Each episodic memory's `currentImportance` decays over time during the slow tick.

### Decay Mechanics

1. **Decay interval:** Memory decay runs every `decayInterval` slow ticks (default: 10, which at 30s slow ticks means roughly every 5 minutes)
2. **Decay formula:** `currentImportance -= decayRate * decayAmount`
   - Standard memories: `decayRate = 1.0`
   - Deep memories (importance >= 9): `decayRate = 0.1` (10x slower decay)
   - Permanent memories (importance 10 / trauma): `decayRate = 0` (never decays)
3. **Pruning:** After decay, memories with `currentImportance` below `pruneThreshold` (default: 0.5) are permanently deleted
4. **Last accessed:** Accessing a memory (retrieving it for context) can refresh its `lastAccessedAt` timestamp

### Memory Categories by Durability

| Category | Base Importance | Decay Rate | Prunable? | Example |
|----------|----------------|------------|-----------|---------|
| Routine | 1-4 | 1.0 (normal) | Yes | "Walked past the market" |
| Notable | 5-7 | 1.0 (normal) | Yes | "Player asked about the quest" |
| Significant | 8 | 1.0 (normal) | Yes | "Completed a trade deal" |
| Deep | 9 | 0.1 (slow) | Yes, but slowly | "Saved the character's life" |
| Permanent/Trauma | 10 | 0 (never) | Never | "Witnessed a devastating loss" |

### Creating Trauma Memories

Trauma memories are permanent and never decay or get pruned. They represent devastating events that haunt the character indefinitely.

```typescript
// Via Engine API
engine.recordTrauma(
  'character-1',
  'Watched helplessly as the village burned to the ground.',
  'Village destruction -- traumatic witness',
  ['trauma', 'village', 'fire']
);
```

## Importance Scoring

When a new episodic memory is created, it receives an importance score from 1 to 10.

### Default Scoring

The built-in importance scorer (`src/memory/ImportanceScorer.ts`) considers:
- Event type and content
- Whether the player was directly involved
- The inherent importance of the triggering event (if `event.importance` is set)

### Plugin Override

Games can provide custom importance scoring via the `GamePlugin.scoreImportance()` method:

```typescript
// In your GamePlugin implementation
scoreImportance(characterId: string, event: GameEvent): number | undefined {
  // Return 1-10 to override default scoring
  // Return undefined to fall back to default
  if (event.type === 'boss_defeated') return 9;
  if (event.type === 'item_pickup') return 2;
  return undefined; // use default
}
```

When a plugin provides `scoreImportance`, the engine creates a composite scorer that tries the plugin first and falls back to the default.

## Episodic Retrieval

When building context for an agent decision, the engine retrieves relevant episodic memories.

### SQL-First Retrieval

The primary retrieval path uses SQL queries against the memory database:

1. Filter by `characterId` and `playerId`
2. Filter by `currentImportance >= importanceThreshold` (default: 3)
3. Sort by a combination of recency and importance
4. Limit to `episodicRetrievalCount` (default: 5) results

### Semantic Fallback

When the SQL-first query returns **fewer than 2 results**, and the `EmbeddingService` is configured, the engine falls back to the `SemanticRetriever`:

1. The current context (trigger event, recent interactions) is embedded into a vector
2. The vector is compared against stored memory embeddings
3. The most semantically relevant memories are returned, regardless of recency

This ensures characters can recall old but relevant memories when a topic resurfaces after a long time.

```
Retrieval Flow:
  SQL query (recency + importance)
       |
       +--> >= 2 results? --> Use SQL results
       |
       +--> < 2 results?  --> SemanticRetriever (embedding-based)
                                  |
                                  v
                              Merge SQL + semantic results (deduplicated)
```

### Context Assembly

The `ContextAssembler` (`src/agent/ContextAssembler.ts`) gathers memories and fits them into the tier-dependent token budget:

- **Active tier:** Up to 800 context tokens, retrieves more memories
- **Background tier:** Up to 400 context tokens, fewer memories
- **Dormant tier:** Up to 250 context tokens, minimal memories

Memories are sorted by importance within the budget. If the budget is exceeded, lower-importance memories are dropped.

## Memory Consolidation

**File:** `src/memory/MemoryConsolidator.ts`

Memory consolidation is a periodic background process that merges similar episodic memories into single records, reducing total memory count while preserving key information.

### Clustering Methods

**Tag-Based Clustering (default):**
- Groups memories that share common tags
- Simple and fast, no external dependencies
- Used when EmbeddingService is not configured

**Semantic Clustering (when EmbeddingService available):**
- Embeds memory content into vectors
- Clusters by cosine similarity
- Produces higher-quality merges that capture thematic connections
- Requires `config.embedding` to be set

### Auto-Consolidation

The engine runs automatic consolidation:
- **Interval:** Every 10 slow ticks (approximately 5 minutes)
- **Batch size:** Up to 5 characters per cycle
- **Threshold:** Only characters with >= 10 episodic memories are candidates
- **Non-blocking:** Runs as a fire-and-forget async operation, does not block the tick loop

### Manual Consolidation

```typescript
// Consolidate memories for a specific character
await engine.consolidateMemories('character-1', 'player-1');
```

### Consolidation Result

```typescript
interface ConsolidationResult {
  mergedCount: number;      // How many memories were merged
  newMemoryId: string;      // ID of the new consolidated memory
  originalIds: string[];    // IDs of the original memories that were merged
}
```

## Summary Auto-Regeneration

Character summaries are periodically regenerated to stay current with the character's evolving history.

### Trigger

Summary regeneration occurs every `summaryRegenerateInterval` slow ticks (default: 50). At the default slow tick rate of 30 seconds, this means roughly every 25 minutes.

### Process

1. Retrieve all current episodic memories for the character
2. Send them to the LLM with a summarization prompt
3. The LLM produces:
   - A narrative `summary` of the character's experiences
   - `relationshipNotes` describing how the character views the player
   - `keyFacts` as a bullet-point list of important information
4. The new summary replaces the old one, with an incremented `version`

### Budget-Aware

Summary generation uses the `heavy` inference tier to ensure quality, since it runs infrequently and the output persists for many decision cycles.

## Configuration Reference

All memory settings are under the `memory` key in the engine configuration:

```json
{
  "memory": {
    "workingMemorySize": 5,
    "episodicRetrievalCount": 5,
    "importanceThreshold": 3,
    "decayInterval": 10,
    "pruneThreshold": 0.5,
    "summaryRegenerateInterval": 50
  }
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `workingMemorySize` | number | 5 | Maximum entries in the working memory ring buffer |
| `episodicRetrievalCount` | number | 5 | Maximum episodic memories retrieved per decision |
| `importanceThreshold` | number | 3 | Minimum `currentImportance` for a memory to be retrieved |
| `decayInterval` | number | 10 | Slow ticks between memory decay passes |
| `pruneThreshold` | number | 0.5 | Memories with `currentImportance` below this are deleted |
| `summaryRegenerateInterval` | number | 50 | Slow ticks between automatic summary regeneration |

## EmbeddingService

**File:** `src/inference/EmbeddingService.ts`

The EmbeddingService is an optional component that provides vector embeddings for text. When enabled, it enhances two subsystems:

1. **SemanticRetriever** -- embedding-based memory retrieval as fallback
2. **MemoryConsolidator** -- semantic clustering for higher-quality memory merges

### Configuration

Add an `embedding` block to your engine config:

```json
{
  "embedding": {
    "type": "vllm",
    "baseUrl": "http://127.0.0.1:8101/v1",
    "models": {
      "heavy": "nomic-embed-text-v1.5",
      "mid": "nomic-embed-text-v1.5",
      "light": "nomic-embed-text-v1.5"
    },
    "maxConcurrency": 4,
    "timeoutMs": 10000
  }
}
```

### Recommended Setup

- Use a dedicated GPU for the embedding model (e.g., GPU 1 with vLLM on port 8101)
- FP16 safetensors format is required for BERT-based embedding models (GGUF is not supported)
- Recommended model: `nomic-embed-text-v1.5` (768 dimensions)

### Without Embeddings

The engine works without the EmbeddingService. Memory retrieval falls back to SQL-only queries, and consolidation uses tag-based clustering. Embeddings are an enhancement, not a requirement.

## Memory in the Decision Pipeline

During each agent decision, memory is assembled as follows:

```
ContextAssembler.buildContext(characterId, playerId, tier)
    |
    +-- Working memory (always included, ring buffer contents)
    |
    +-- Episodic memories
    |     |
    |     +-- SQL query: recent + important
    |     |
    |     +-- (if < 2 results) SemanticRetriever fallback
    |     |
    |     +-- Sort by importance, fit to token budget
    |
    +-- Character summary (if exists)
    |
    +-- Extension data (emotions, goals, relationships, needs, etc.)
    |     |
    |     +-- Only injected if token budget allows
    |
    v
PromptBuilder.build(context)
    |
    +-- Structured XML sections
    +-- [Identity] [Summary] [Memories] [Situation] [Tools] [Hints]
    |
    v
  System prompt + user message --> LLM
```

The token budget is tier-dependent:
- **Active:** 800 tokens -- room for all sections and multiple memories
- **Background:** 400 tokens -- summary + 1-2 key memories + minimal extensions
- **Dormant:** 250 tokens -- summary only + most critical context
